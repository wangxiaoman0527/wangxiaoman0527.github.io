🚀 GitHub Actions + Hexo 自动化部署：从零开始到完美避坑指南

兄弟们，我终于把Hexo博客的自动化部署搞定了！本来以为就是点几下鼠标的事，结果差点被SSH密钥和404错误给送走……😅

现在我的博客已经实现了「写文章 -> git push -> 自动部署」的全流程自动化，爽到飞起！今天就把我从零开始踩坑的经验分享给大家，保证说人话，不整那些看不懂的术语。

🤔 为什么要自动化部署？

想象一下这个场景：

  1. 你在本地写了篇超棒的文章
  2. hexo clean && hexo g && hexo d 三连
  3. 喝杯咖啡，博客就更新了

是不是很丝滑？但前提是你要在本地操作。如果你换电脑了，或者想在手机上写文章（虽然不太现实），就傻眼了。

自动部署就是为了解决这个问题：无论在哪，只要把文章推送到 GitHub，它就会自动帮你构建和部署。

📦 准备工作

在开始之前，你需要：

  1. 一个 Hexo 博客（废话）
  2. 两个 GitHub 仓库：
    • 你的用户名.github.io（存放生成的静态文件，GitHub Pages 会自动从这个仓库部署）
    • blog-source（存放 Hexo 源码，可以随便取名，建议设为私有）

重点来了：那个 -N "" 意思是「不要密码」。如果你设置了密码，GitHub Actions 运行时没法输入密码，就会认证失败。

步骤 2：配置公钥到部署仓库

  1. 查看公钥：cat ~/.ssh/github-hexo-deploy.pub
  2. 复制全部内容
  3. 进入 你的用户名.github.io 仓库
  4. Settings → Deploy keys → Add deploy key
  5. 随便起个名，粘贴公钥
  6. 最重要的一步：勾选 ✅ Allow write access（不勾选就是只读，部署会失败）

步骤 3：配置私钥到源码仓库

  1. 查看私钥：cat ~/.ssh/github-hexo-deploy
  2. 复制全部内容（包括 -----BEGIN... 和 -----END... 那几行）
  3. 进入你的源码仓库
  4. Settings → Secrets and variables → Actions → New repository secret
  5. 名字填 HEXO_DEPLOY_KEY，值粘贴私钥

🧐 部署成功但网站空白？看这里！

如果你部署成功了但访问网站是一片空白，大概率是以下原因：

1. 🚫 主题下的 .git 文件夹冲突（最大坑！）

问题：如果你是用 git clone 下载的主题，那么主题目录下会有一个隐藏的 .git 文件夹。这会让Git将主题视为一个子模块，而不是你项目的一部分。导致自动化部署时，主题文件根本不会被上传！

解决：

  1. 找到你的主题目录（比如 themes/next/）。
  2. 删除里面的 .git 文件夹。
    • 可以在终端里进入主题目录，运行：rm -rf .git
  3. 然后回到博客根目录，重新提交所有文件：

2. 🛑 Jekyll 的“无视”行为

问题：GitHub Pages 默认使用 Jekyll 构建网站。而 Jekyll 会忽略所有以下划线 _ 开头的文件和文件夹。巧了，Hexo的很多主题源文件正好就是以 _ 开头的（如 _config.yml, _source/），所以它们全被无视了！

解决：
在你的Hexo博客的 source 目录下，手动创建一个名为 .nojekyll 的空文件（没有后缀名）。这个文件会告诉GitHub：“别用你的Jekyll处理我的网站，直接用我生成好的文件就行”。

💡 常见翻车点

  1. SSH 密钥有密码：自动化流程没法输入密码，必须用无密码密钥
  2. 没勾选 Allow write access：公钥只有读权限，部署会失败
  3. 用错了仓库地址：一定要用 SSH 格式的地址
  4. 私钥没复制全：包括 -----BEGIN... 和 -----END... 都要复制
  5. 主题下有 .git 文件夹：会导致主题文件不被上传
  6. 没有 .nojekyll 文件：导致主题文件被 Jekyll 忽略

现在，我的写作流程变成了：

  1. 本地写文章
  2. git add . && git commit -m "又一篇大作" && git push
  3. 收工！

GitHub Actions 会自动帮我完成构建和部署，再也不用记那三条 hexo 命令了。
